# 股察小助手 - 个人A股观察工具产品需求文档（PRD）

**文档版本**：1.0  
**创建日期**：2026-04-02  
**产品定位**：个人与家人专用的A股观察小工具  
**技术定位**：极简Web应用，无后端或极简后端

## A. 原PRD在新前提下的主要问题

原PRD设计过于复杂，主要问题：
1. **功能过度**：面向商业产品设计，包含全市场列表、复杂预警、多维分析
2. **架构过重**：需要代理服务、定时任务、复杂数据存储
3. **维护成本高**：全市场数据更新和维护负担重
4. **开发周期长**：8周计划不适合个人开发

## B. 优化后的产品定位

**股察小助手** - 个人与家人专用的A股观察工具，专注少量关注股票的价格、股息率查看与简单跟踪。

**核心原则**：
- 极简：一个页面解决核心需求
- 稳定：数据源简单可靠
- 易维护：纯前端或极简后端
- 够用：功能满足日常观察，不追求全面

## C. 优化后的PRD

### 1. 产品目标
为个人投资者提供简单的A股股价与股息率观察工具，支持手动查询关注股票并加入自选列表进行跟踪。

### 2. 目标用户
- 本人及家人（1-5人）
- 关注10-50只股票的长期表现
- 能接受简单界面，不追求专业软件体验

### 3. 使用场景
1. **日常查看**：打开网页，查看自选股最新价格和股息率
2. **新增关注**：手动输入股票代码，查看详情后加入自选
3. **简单对比**：切换查看不同股票的详细信息
4. **长期跟踪**：定期查看持股表现，关注股息变化

### 4. 功能范围

#### V1 必做功能（核心）
1. **手动股票查询**
   - 输入框支持股票代码（000001）或名称（平安银行）
   - 调用公开API获取实时数据
   - 显示：价格、涨跌幅、股息率、市值

2. **自选股管理**
   - 添加/删除自选股（localStorage存储）
   - 自选股列表展示（最新价、涨跌幅、股息率）
   - 点击自选股查看详情

3. **股票详情展示**
   - 核心数据展示区
   - 简单股息历史（最近3年）
   - 基础基本面（PE、PB）

#### V1 可选功能（有时间就做）
1. 简单价格走势图（30天，Chart.js轻量版）
2. 浏览器本地提醒（价格超过阈值）
3. 数据手动刷新按钮

#### 明确延期到后续版本
1. 全市场股票列表浏览
2. 复杂图表分析（K线、多周期）
3. 自动定时数据更新
4. 多设备数据同步
5. 数据导出/导入
6. 复杂预警系统

- 响应式但优先桌面体验
- 操作集中在下部

### 6. 数据方案

#### 数据源选择
**主要API**：腾讯财经 `https://qt.gtimg.cn/q=sh600000,sz000001`
- 优点：简单直接、CORS限制较少
- 数据：价格、涨跌幅、成交量等实时数据

**股息和基本面数据**：
- 方案A：手动维护本地JSON文件（每季度更新一次）
- 方案B：使用东方财富简单API（需处理CORS）
- **推荐方案A**：维护最简单

#### 数据更新策略
- **实时数据**：用户查询时获取，页面加载时获取自选股数据
- **股息数据**：本地JSON文件，每季度手动更新
- **无自动定时任务**：避免维护负担

#### 本地存储
- 使用`localStorage`存储自选股列表
- 数据格式：`{code: "000001", name: "平安银行", addedAt: "2026-04-02"}`
- 容量：最多50只股票，完全够用

### 7. 技术方案

#### 技术栈推荐
```
前端：Vue 3 + Composition API（或原生JavaScript）
CSS：纯CSS或Tailwind CSS
图表：Chart.js轻量版（可选）
无框架：避免React等较重方案
无状态管理：避免Redux/Zustand等复杂度
```

#### 架构选择（二选一）
**方案A（纯前端）**：
- 直接调用公开API（处理CORS）
- GitHub Pages静态部署
- 完全无后端，零维护成本

**方案B（极简后端）**：
- 简单Node.js代理（仅解决CORS）
- 部署到Vercel（Serverless）
- 每月手动更新一次基本面数据

**推荐方案A**，优先尝试纯前端。

#### 部署方案
- 静态托管：GitHub Pages（免费、简单、自动HTTPS）
- 域名：可不用，或使用免费子域名
- 无需数据库，无需服务器维护

### 8. 非功能要求

#### 性能目标
- 页面加载：< 2秒（无大量资源）
- 查询响应：< 3秒（受API响应影响）
- 本地操作：即时响应

#### 稳定性
- 核心查询功能可用性 > 95%
- 数据错误时有明确提示
- 网络异常时显示友好信息

#### 维护性
- 代码总量 < 1000行
- 文件数量 < 10个
- 无数据库维护需求
- 配置项 < 5个

#### 浏览器兼容
- 现代浏览器：Chrome、Firefox、Edge最新版
- 不考虑IE等老旧浏览器

### 10. 设计取舍与风险说明

#### 为什么V1不做全市场？
- **必要性低**：个人关注股票有限（10-50只）
- **成本高**：数据获取和维护成本大幅增加
- **复杂度**：界面和功能复杂度提升
- **后续可加**：需要时可逐步添加，不作为核心

#### 为什么V1不做复杂提醒？
- **使用频率**：个人使用频率不高，手动查看即可
- **权限麻烦**：浏览器通知需要用户授权，体验下降
- **维护负担**：定时任务增加系统复杂度
- **后续可简**：可做简单本地提醒，不做推送系统

#### 为什么V1不做过多图表和分析？
- **核心需求**：是"查看"而非"分析"
- **资源负载**：复杂图表库增加页面大小
- **专业补充**：数据分析可用专业工具（同花顺等）完成
- **渐进增强**：后续可添加简单图表，不追求全面

#### 为什么先做手动查询？
- **定位准确**：符合"小工具"用完即走的定位
- **避免问题**：避免股票池同步和自动更新问题
- **数据及时**：用户主动查询时数据最新
- **实现简单**：无定时任务，无后台复杂度

#### 技术风险与应对
1. **API稳定性**：免费API可能变动
   - 应对：选择多个备用API，代码做好兼容

2. **CORS限制**：浏览器可能阻止直接调用
   - 应对：准备极简代理方案（Node.js + Vercel）

3. **数据准确性**：免费数据可能有误差
   - 应对：明确告知"数据仅供参考"，关键数据手动核对

#### 维护风险
1. **数据源变更**：API接口变化
   - 应对：封装数据获取逻辑，变化时只需修改一处

2. **长期维护**：代码要简单到1年后还能看懂
   - 应对：避免复杂框架，写清晰注释

## E. 给开发者的最终建议

### 1. 开发优先级顺序
```
第一（必须完成）：
1. 搜索查询功能 ← 最核心
2. 自选股本地存储 ← 次核心  
3. 基础数据展示 ← 核心体验

第二（有时间就做）：
4. 简单图表（Chart.js轻量版）
5. 本地提醒（Notification API）

第三（完善阶段）：
6. 错误处理优化
7. 移动端适配
8. 部署上线
```

### 2. "足够好"的发布标准
- 能查询股票 ✅
- 能加自选股 ✅  
- 能看价格和股息率 ✅
- 界面清晰不丑 ✅
- 无重大bug ✅
- 本人和家人会用 ✅

### 3. 技术选型避坑指南
**必须避免**：
- ❌ React（相对重）
- ❌ 状态管理库（Redux/Zustand）
- ❌ 复杂图表库（ECharts等）
- ❌ 后端数据库
- ❌ 用户认证系统

**推荐使用**：
- ✅ Vue 3 或 原生JS
- ✅ localStorage
- ✅ Chart.js轻量版（如需图表）
- ✅ GitHub Pages部署

### 4. 后续迭代原则
如果V1用得好，后续考虑：
1. 添加简单数据导出（CSV）
2. 增加3-5只股票对比
3. 添加股息日历提醒
4. 优化图表体验

**关键问题**：每次新增功能都要问"是否真的需要？会不会增加维护负担？"

### 5. 最重要原则
**为5年后的自己开发**：
- 代码简单到1年后还能看懂
- 架构简单到不需要文档也能维护
- 依赖少到不会因版本过时而无法运行
- 功能实用到真的会经常使用
